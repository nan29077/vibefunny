## 2. 주요 플로우 (화면 순서·흐름)

### 2-1. 크리에이터

**(a) 가입 ~ 활성화** (`auth-actions.ts`, `payment/effects.ts`)
1. `/signup`에서 역할 선택, 추천코드(선택) 입력 → 이메일 중복·추천코드 검증
2. 가입비 정책 ON이면 `payment(signup_fee, pending)` 생성 → `/payment/activate`로 이동, Mock 결제 성공 시 `status: active` 전환
3. 추천 제도 ON이면 추천인에게 `referral_reward(PENDING)` + 지갑 보류 적립 생성
4. 가입비 OFF면 즉시 활성화 후 `/creator` 이동

**(b) 캠페인 참여 — 배포/영상제작** (`creator-campaigns-ui.tsx`, `submission-actions.ts`)
- 참여 가능 캠페인 목록 → 상세 → **참여 유형 선택**: `deploy`(배포) 또는 `video_production`(영상제작)
- 상태전이(`campaign_participations`): `applied → accepted →` (영상제작) `video_submitted → video_approved →` (배포) `deploy_submitted → deploy_approved → completed`
- 영상제작자: 원본 영상(파일 Base64/URL) 제출 → 광고주/관리자 승인
- 배포자: SNS 게시물 URL 제출 → 승인 시 배포 단가 지급
- 승인 시점에 `wallet_transactions(campaign_reward, available)` 적립

**(c) 영상판매(마켓)** (`creator/videos`, `video-actions.ts`)
1. `/creator/videos/new`에서 영상 등록(제목·플랫폼·길이·가격·저작권 확인)
2. `video_auto_approve`에 따라 `available`(자동) 또는 `pending_review`(관리자 승인 대기)
3. (선택) **바이브포터 노출 신청** → `vibeporter_enabled=true` → 관리자 `/admin/vibeporter`에서 승인 → `vibeporter_approved=true`
4. 구매 발생 시 `status: sold`(1회 한정), 판매자에게 수수료(`video_sale_platform_fee_rate`) 차감 후 지갑 적립

**(d) 쇼츠 커머스** (`shorts-commerce-panel.tsx`, `commerce-actions.ts`)
1. `/creator/social`에서 운영 유튜브 채널 등록
2. `/creator/shorts-commerce`에서 쇼츠 URL 입력 + **영상 출처**(self / vibeporter) 선택
3. 키워드·카테고리 기반 상품 추천(`recommend.ts`) → 상품 연동(`linked_product_ids`)
4. 주문 발생 시 `product_orders` 생성, 크리에이터 수수료(`commission_rate`) 적립

**(e) 영상판매 vs 캠페인 영상제작 구분**
- 영상판매 = 내가 만든 기성 영상을 **마켓에 판매**(`videos`)
- 캠페인 영상제작 = 광고주 브리프대로 **주문제작**(`campaign_participations.participation_type="video_production"`)

**(f) SNS/숏폼 채널** (`creator/social`, `creator/community`)
- 배포용 SNS 계정(`social_accounts`)·운영 유튜브 채널(`creator_youtube_channels`) 등록(인증상태는 unverified 기본)
- 맞구독 디렉터리(`/creator/community`)는 DB 쓰기 없는 **순수 조회/외부링크**

**(g) 정산/출금** (`payout-actions.ts`)
- `/creator/points`에서 가용잔액 내 출금 신청 → `payout_request(requested)` + 지갑 차감 → 관리자 승인→지급(`paid`)

### 2-2. 광고주 (`campaign-actions.ts`, `point-actions.ts`, `queries.ts`)
1. `/advertiser/points`에서 포인트 충전(Mock 결제)
2. `/advertiser/campaigns/new`에서 캠페인 등록 — 유형(`create_and_distribute` / `distribute_*` / `create_only`), 플랫폼별 배포 건수, **영상 길이 구간(15/30/60/90s)** 선택
3. 서버가 단가표 기준 **비용 재계산**(클라이언트 금액 불신): 배포비(`distribution_rates`) + 제작비(`video_pricing_tiers` 또는 `extra_creation_*`)

6. 크리에이터 참여·증빙 → 광고주/관리자 증빙 승인 → 크리에이터 정산
7. 실행사–대행사: 대행사 집행 시 실행사 수수료 자동 생성(기준: `advertiser_commission_basis`)

### 2-3. 최고관리자 (`settings-actions.ts`, `admin-actions.ts`, `commerce-actions.ts`)
- **정책 설정**(`/admin/settings`): 가입비/구독료/제작단가(크리에이터 지급+광고주 청구)/배포단가/판매수수료/추천제도/실행사 수수료를 즉시 변경 → 이후 생성 거래부터 반영

- 발주/배송(`/admin/products/orders`): 주문 상태·택배사·운송장 입력

## 3. 바이브포터 · AI스토리 연동 점검

### 3-1. 바이브포터 — **실연동 아님 (앱 내부 기능)**
- 외부 "바이브포터" 서버/API 호출 코드는 **없음**. 전부 `data/db.json` 내부에서 처리된다.
- 구성: `videos`의 `vibeporter_enabled / vibeporter_approved / vibeporter_price` 플래그 + `vibeporter_requests` 컬렉션(더미) + 관리자 화면(`/admin/vibeporter`, `api/admin/vibeporter/*`).
- 즉, "바이브포터 연동"은 실제로는 **자사 영상마켓의 노출 승인 워크플로우 + 쇼츠 영상 출처 라벨(self/vibeporter)**일 뿐, 별도 외부 플랫폼과의 통신은 없다.
- **결론: 외부 연동 없음. 추후 별도 바이브포터 서비스가 생긴다면 API 신설 필요.**

### 3-2. AI스토리 — **부분 실연동(로컬 파일 공유) + 설정 UI는 미사용 자리**
세 갈래로 나뉜다.

1. **실제 동작하는 부분 — 형제 폴더 파일 공유**
   `src/app/api/story-commissions/route.ts`, `.../[id]/route.ts`가 `process.cwd()/../AI스토리/data/story_requests.json`(= `E:\프로젝트\AI스토리\data\story_requests.json`)을 **직접 read/write**한다. GET으로 승인된 의뢰만 노출, PATCH로 `join`(진행중)·`complete`(완료) 상태를 AI스토리 파일에 되써준다.
   → **같은 PC에 `E:\프로젝트\AI스토리` 프로젝트가 형제 폴더로 존재해야** 동작하는 **로컬 파일 연동**. 네트워크 API가 아니다. (이번 점검 환경에는 해당 폴더가 없어 빈 배열 반환.)

2. **자리(설정 UI)만 있고 실제로는 미사용**
   `/admin/ai-story`의 연동 설정 폼(`api_base`, `api_key`, `webhook_secret`, `auto_import`)은 값이 `db.settings.ai_story`에 **저장만** 되고, 이 값을 사용해 외부 API를 호출하거나 webhook을 수신하는 코드는 **전혀 없다**(grep 확인). "auto_import(자동 수신)"도 미구현. 화면의 "실제 연동 활성화" 배지는 키 입력 여부만 보고 표시할 뿐 기능 없음.

3. **동화 제작 의뢰 캠페인** — db.json 내부 더미
   `campaign_type="story_creation"`, `source_type="ai_story"`, `story_brief`(주인공/테마/그림체/목소리 파일 등) 형태로 `ad_campaigns`에 들어있고, 크리에이터 캠페인 화면에 노출된다. `src/lib/db.ts`가 더미 동화 캠페인(`camp_story_005` 등)과 목소리 파일 URL을 **자동 주입**한다(soundhelix 샘플 mp3). 즉 **데모 데이터**이며 AI스토리 앱과의 실시간 연동 결과가 아니다.

- `creator/story-commissions` 경로는 `/creator/campaigns`로 **리다이렉트**(별도 메뉴 통합됨).
- **결론: AI스토리는 "같은 PC의 형제 폴더 JSON 파일 공유"만 실동작. API/webhook 기반 실연동은 설정 UI(자리)만 있고 미구현. 동화 캠페인은 데모 데이터.**

### 4-2. 깨진 부분 / 주의점
1. **API 라우트 권한 검증 누락(보안)** — `src/app/api/` 13개 라우트 중 서버측 권한체크(`requireRole/requireUser`)가 있는 것은 **1개뿐**. 예: `api/admin/vibeporter/approve`는 인증 없이 누구나 영상 승인/노출 가능, 크리에이터 참여/제출 API들이 `creator_id`를 **클라이언트 입력값으로 신뢰**. → 운영 전 반드시 인증/인가 추가 필요. (Server Actions 쪽은 `requireRole`로 보호됨)
2. **buyer 역할 잔재** — `app/buyer/*` 페이지가 전부 `redirect("/")`. 동작상 문제는 없으나 죽은 코드. README의 buyer 설명도 outdated.
3. **AI스토리 설정 UI가 오해 소지** — "실제 연동 활성화" 표시가 실제 기능과 무관(미구현). 운영 메시지 정리 필요.
4. **AI스토리 파일 연동의 경로 의존성** — `../AI스토리` 형제 폴더 하드코딩. 폴더 없으면 조용히 빈 배열. 배포 환경(서버)에서는 동작 불가.
5. 잔재 파일: 루트 `_restore_쇼츠커머스이전/`(백업 zip 등), `tsconfig.tsbuildinfo` 등은 정리 대상(기능 무관).

### 4-3. mock으로만 동작하는 부분
- **결제 전반**: 가입비/구독/포인트충전/영상구매 모두 `MockPaymentProvider`로 항상 성공 처리(`PAYMENT_PROVIDER=mock`).
- **카페24**: 키 미입력 시 `MOCK-xxxx` 가짜 상품번호 반환(실제 등록 안 됨). 키 입력 시에만 실 API 1엔드포인트 호출.
- **다운로드 URL**: 영상 구매 후 다운로드는 서명 URL "모사"(실 스토리지 아님).
- **SNS 계정 인증**: `verified_status`는 기본 unverified, 실제 플랫폼 인증 로직 없음.

### 4-4. DB/백엔드 상태
- **localStorage 사용 0건**(전수 grep). 클라이언트 저장 아님.
- 데이터 레이어 = **서버사이드 단일 JSON 파일** `data/db.json`을 `fs`로 read/write. 트랜잭션 헬퍼 `tx()`는 매 호출 시 파일 전체를 읽고→변경→다시 씀(파일 락 없음 → 동시성 취약, 단일 사용자 데모엔 충분).
- 인증 = **서명 쿠키(HMAC, `vf_session`)** 기반 Mock Auth. 비밀번호는 해시 저장(`password_hash`), SNS 비밀번호는 미저장(플랫폼/계정명/URL/인증상태만).
- Supabase용 마이그레이션 SQL(`supabase/migrations/0001_init.sql` 21테이블 + `0002_rls.sql`)과 `seed.sql`은 준비돼 있으나 **앱은 아직 로컬 JSON만 사용**(`DATA_DRIVER=supabase` 구현 미연결).
- 금액은 모두 KRW 정수, 서버에서만 계산. 포인트/지갑은 ledger(`point_transactions`/`wallet_transactions`) 경유. 결제효과(`applyPaymentEffects`)는 멱등.

## 결론 / 우선순위 제언

1. **운영 전 필수(보안/정합성)**: API 라우트 인증·인가 추가, 로컬 JSON → Supabase 이관(동시성).
2. **수익 연결(결제)**: Toss 등 실 PG 연동.
3. **연동 완성**: 카페24 OAuth 자동화 + 주문/이미지 동기화, AI스토리 HTTP/Webhook 실연동.
4. **자산/증빙**: 스토리지 + SNS API로 업로드·성과 자동화.
5. **정리**: buyer 잔재·`_restore` 백업·AI스토리 "연동 활성화" 오해 문구 정리.

> 전반적으로 도메인 모델(21+테이블)과 화면·서버액션은 충실히 구현된 **데모 가능한 MVP**다. 다만 "연동"으로 표기된 카페24/바이브포터/AI스토리는 **카페24만 부분 실연동, 나머지는 앱 내부 기능 또는 로컬 파일/자리(placeholder)** 수준이므로, 외부 서비스화에는 위 표의 API 작업이 필요하다.
